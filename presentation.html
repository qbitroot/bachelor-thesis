Téma práce

Cieľ: Návrh a implementácia fuzzera pre protokol WebTransport over HTTP/3 — testovanie robustnosti a bezpečnosti rôznych serverových implementácií.

  • WebTransport = moderný protokol nad QUIC (RFC 9000) / HTTP/3 (RFC 9114).
  • Fuzzing = automatizované posielanie malformovaných vstupov na odhalenie chýb.
  • Cieľové implementácie: Rust (wtransport), Go (webtransport-go), Python (aioquic).

Čo je hotové

  • Naštudovaná literatúra done
    RFC 9000 (QUIC), RFC 9114 (HTTP/3), RFC 9297 (Capsule Protocol), draft-ietf-webtrans-http3 (ešte stále draft, nie RFC)
  • Fuzzer implementovaný done
    Python + Boofuzz + aioquic. 3 fuzzing modes, tisíce test casov.
  • Echo servery inštrumentované done
    Rust (wtransport), Python (aioquic) a Go (webtransport-go) — všetky s WTFUZZ structured log formátom.
  • Bug v Rust knižnici objavený done
    Reachable assertion panic vo wtransport.
  • Log analýza pipeline done
    SQLite DB + log grouping podľa SHA-256 fingerprint serverového výstupu.

Fuzzing modes

oneshot — 1 malformed capsule per session.
Fuzz: type, length, payload (boundary values, overlong VarInts, cross-layer type confusion).
multistep — multi-step scenáre na živej session.
Scenáre × mutácie (permutácie, duplikácie, vynechania, rapid-fire, zakázané capsule injection).
scenario-lastfuzz — scenár ako prefix, posledný step fuzzed.
Kombinuje state-richness multistep + coverage oneshot capsúl.
all — všetky modes v jednom behu, deduplikované test cases.

Scope of fuzzing

Každý test case zaznamenáva presne, čo bolo odoslané serveru — formát action(hex): 

capsule(800078ae00)       -- send data on CONNECT capsule control stream
bidi(48454c4c4f)          -- open bidi WT stream, send data, await echo
uni(554e49)               -- open uni WT stream, send data
datagram(4447)            -- QUIC datagram

CONNECT stream = obyčajný bidirectional QUIC stream, ktorý sa používa na session setup (HTTP CONNECT) a potom slúži ako control channel pre capsules (RFC 9297). Nie je to špeciálna vrstva — je to ten istý QUIC stream, na ktorom posielame fuzzed data.

WT streams/datagrams = ďalšie QUIC streams/datagrams asociované so session cez Session ID. Fuzzer ich otvára, posiela dáta a sleduje echo response.

Fuzzing methodology

Health Check

Po každom test case fuzzer otvorí nový bidi stream a pošle echo probe. Ak server neodpovie do 1s — logged ako failure. Detekuje crashe, memory corruption a state confusion.

Log Grouping

Serverový output je SHA-256 fingerprintovaný. Test cases s identickým outputom zdieľajú jednu log_group — grouping podľa failure mode. Eliminuje duplicity, umožňuje rýchlu analýzu.

WTFUZZ formát: Inštrumentované servery emitujú WTFUZZ|<conn>|EVENT|key=val na stdout. Offline korelácie cez analyze_logs.py.

Finding: wtransport (Rust)

Reachable Assertion Panic

github.com/BiagioFesta/wtransport

  • Príčina: Capsule::with_frame() má hard assert!(FrameKind::Data), ale validate_frame() prepustí aj Headers a Exercise frames — assertion panics.
  • Dopad: Worker thread panic. V Rust-e by knižnica nemala nikdy panikovať — panic v async tasku spôsobí tichý pád handlera.
  • Fix: Guard frame.kind() check pred volaním Capsule::with_frame(), skip non-Data frames.

Problémy a výzvy

  • Kombinatorická explózia
    Plný priestor test casov by vyžadoval behy potenciálne trvajúce viacero dní. Umelo zužujeme na cases ktoré majú najvyššiu pravdepodobnosť odhaliť bugs (boundary values, zakázané capsule types, state conflicts).
  • WebTransport je stále draft
    Špecifikácia (draft-ietf-webtrans-http3) sa aktívne mení počas písania práce — implementácie sú na rôznych draft verziách (draft-03, draft-09, draft-15). Riešené tým, že podľa RFC 9297 musia servery ignorovať neznáme capsule types — ak namiesto toho crashnú, je to bug.
  • Go server: chýbajúce stream ID
    Knižnica webtransport-go neexponuje QUIC stream ID cez svoje API, takže WTFUZZ logy z Go servera neobsahujú stream_id. Neovplyvňuje fuzzing, ale sťažuje koreláciu logov.
  • Formalizácia v texte
    Implementácia je hotová, ale premietnutie do formálneho popisu v bakalárskej práci vyžaduje čas a iterácie.
